Overview
The pipeline answers a single question: what distress signals were active on fulfilled properties at the moment they were delivered — and subsequently sold?
This guide walks through every step required to produce a Fulfillment Distress Analysis report for any 8020REI client. Steps 1–3 are automated Python scripts run by the team member. Step 4 is a Claude Code task that reads the prepared data, runs the analysis, and generates the final HTML + PDF report.

What Claude Code does: It reads the two output files from the pipeline, computes all distress signal statistics, constructs the full multi-page report (HTML), and exports it as a print-ready PDF. No manual analysis or formatting is needed.

Prerequisites

Software

  • Python 3.9+ — required to run the pipeline scripts
  • Claude Code CLI — for Step 4 report generation
  • Playwright + Chromium — for PDF export (auto-installed)
# First-time setup pip install -r requirements.txt pip install playwright python -m playwright install chromium

Source Files Needed

  • Monthly fulfillment XLSX files — one per delivery month, placed in 01_fulfillment_merger/input/. Filename must start with YYYY-MM-DD so the pipeline can extract the period.
  • Domain COO XLSX file(s) — full platform snapshot with distress columns, placed in 02_data_processing/input/. Used for both signal universe counts (Step 2) and the sales index (Step 3). One or multiple files are accepted and merged automatically.
Mandatory Reading Before First Run
  • fulfillment_distress_protocol (4).txt — defines analysis window, data encoding, report structure, and standard recommendations
  • analysis_notes.md — documents every non-obvious data decision (signal reading rule, ABSENTEE encoding, join keys, MFR fallback)

Data Preparation
Place the source files in two input folders, then run one command — the pipeline produces the two files that the report generator needs.

Folder Structure

historical_distress_report/ main.py ← run this for steps 1–2 requirements.txt CLAUDE.md analysis_notes.md 01_fulfillment_merger/ input/ ← monthly XLSX files here (YYYY-MM-DD name) output/ ← ✓ {client}_Merged_Fulfillments.parquet merge.py 02_data_processing/ input/ ← full domain COO XLSX file(s) here output/ ← ✓ Distress Overview.xlsx analyze.py 03_generate_report/ generate.py ← run separately after main.py

Steps 1 – 2: Run the Pipeline

Drop the source files in the correct input folders, then run a single command from the pipeline root:

python main.py "Client Name"

This executes two steps automatically:

Step 1
Merge Monthly Fulfillments
Reads all XLSX files from 01_fulfillment_merger/input/ and merges them into a single parquet file. The PERIOD column (e.g. 2025-08) is extracted from each filename — filenames must start with YYYY-MM-DD. Output: {client}_Merged_Fulfillments.parquet
Step 2
Compute Distress Universe
Reads the full domain COO XLSX file(s) from 02_data_processing/input/ and counts active distress signals per county across all recommended properties. Output: Distress Overview.xlsx — used for the Distress Universe table on Page 4 of the report.

Report Generation
Once both output files exist, run the report generator — it analyses the data and produces the client-ready HTML and PDF automatically.
The Command (Step 3)

From the pipeline root directory, run:

python 03_generate_report/generate.py "Client Name" clientslug --window YYYY-MM YYYY-MM

Arguments: client name in quotes → lowercase slug (used in filename) → --window start month end month. The window must match the date range of the fulfillment files in 01_fulfillment_merger/input/.

Examples:

# SBD Housing — 6-month window Aug 2025–Jan 2026 python 03_generate_report/generate.py "SBD Housing" sbdhousing --window 2025-08 2026-01 # FreedomREI — 6-month window Jul–Dec 2025 python 03_generate_report/generate.py "FreedomREI" freedomrei --window 2025-07 2025-12
Outputs
  • HTML report — YYYY-MM-distress-analysis-{slug}.html — saved to pipeline root
  • PDF report — YYYY-MM-distress-analysis-{slug}.pdf — this is the file sent to the client

Logos are embedded as base64 — no HTTP server required to view or convert.

Analysis Window Rule

The window matches the date range of the fulfillment files. Read the earliest and latest YYYY-MM from the filenames. Standard cadence = 6 months ending 3 months before the report date.

What generate.py Does Automatically
  1. Loads the merged fulfillment parquet from 01_fulfillment_merger/output/
  2. Loads the domain COO from 02_data_processing/input/ (uses parquet cache on repeat runs); joins on FOLIO to identify sold properties
  3. Reads signals from the row where PERIOD == MFR month; uses closest period if MFR is outside the data window
  4. Applies ABSENTEE encoding: values 1 and 2 both count as active; all other signals active when value = 1
  5. Counts active signals per property; computes percentages, stacking distribution, county breakdown, buyer type, and monthly sale volume
  6. Reads 02_data_processing/output/Distress Overview.xlsx for the Distress Universe table (Page 4)
  7. Embeds logos, CSS, and all chart data inline — single self-contained HTML file
  8. Exports PDF via Playwright/Chromium (headless)

If Playwright is not installed: HTML is still generated; PDF export is skipped with a warning. Fix: pip install playwright && python -m playwright install chromium

Matched CountiesReport Structure

County count = counties with at least one matched sold property during the window. Counties with zero sales still appear in Page 3 with an amber flag box.

Standard Recommendations (always include)

Recommendation 1
Rapid Response
Parallel same-day outreach channel. Frame as the answer to the speed-to-contact gap. Always include, regardless of signal mix.
Recommendation 2
Repeat Analysis in 6 Months
One dataset is an observation. Two is a pattern. Always recommend the follow-up cycle.
Recommendation 3
Niche Lists (conditional on VA capacity)
Track A: High Equity + Absentee — large pool, validated signal.
Track B: Divorce + Pre-Foreclosure — small universe, high urgency, low competition.
Key Methodology Rules
  • Inclusion: A property is included if its LAST SALE DATE falls within the analysis window — regardless of when it was recommended
  • Signal reading: Read distress columns from the row where PERIOD == MFR month
  • ABSENTEE: Values 1 (in-state) and 2 (out-of-state) both count as active
  • Annex dates: Show month + year only (e.g., "Oct 2025") — never exact day
  • Zero-signal properties: Never mention their count in the report narrative
